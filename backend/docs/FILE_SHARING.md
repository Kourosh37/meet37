# Browser-To-Browser File Sharing

File sharing is designed as browser-to-browser transfer. The backend coordinates file-transfer signaling but does not store, proxy, scan, or stream file bytes.

## Current Backend Scope

Implemented backend behavior:

- Relays file-transfer signaling messages over WebSocket:
  - `file-offer`
  - `file-answer`
  - `file-candidate`
- Preserves sender and target peer IDs.
- Persists transfer metadata so the frontend can show recent offers/answers after reconnect.
- Keeps file payload bytes out of the server.

Intentionally outside backend scope:

- File storage.
- Upload endpoint.
- Download endpoint.
- Virus scanning.
- Server-side file size enforcement.

## Why Browser-To-Browser

Browser-to-browser file transfer keeps server bandwidth and storage low.

Default flow:

```text
Browser A -- file metadata via backend WebSocket --> Browser B
Browser A -- file bytes via WebRTC DataChannel ----> Browser B
```

The backend sees small JSON messages. The file itself goes over WebRTC.

## Recommended Transport

Use WebRTC `RTCDataChannel` for file bytes.

Recommended settings:

```js
const channel = pc.createDataChannel("file", {
  ordered: true
});
```

For large files, send chunks and apply backpressure using:

```js
channel.bufferedAmount
channel.bufferedAmountLowThreshold
channel.onbufferedamountlow
```

## Signaling Messages

The backend relays file messages to `to`.

### `file-offer`

Sender to receiver:

```json
{
  "type": "file-offer",
  "to": "receiver-peer-id",
  "payload": {
    "file_id": "uuid",
    "name": "report.pdf",
    "size": 1234567,
    "mime": "application/pdf",
    "channel": "datachannel-label-or-id"
  }
}
```

Recommended payload fields:

- `file_id`: unique transfer ID generated by sender.
- `name`: display filename.
- `size`: file size in bytes.
- `mime`: browser-provided MIME type.
- `channel`: data channel label or transfer channel identifier.

The backend persists `file_id`, name, size, MIME type, sender peer, target peer, status, and timestamp. It does not inspect or validate the file bytes.

### `file-answer`

Receiver to sender:

```json
{
  "type": "file-answer",
  "to": "sender-peer-id",
  "payload": {
    "file_id": "uuid",
    "accepted": true
  }
}
```

Reject example:

```json
{
  "type": "file-answer",
  "to": "sender-peer-id",
  "payload": {
    "file_id": "uuid",
    "accepted": false,
    "reason": "File too large"
  }
}
```

### `file-candidate`

Reserved for file-specific negotiation metadata if the frontend creates separate peer connections or data channels for file transfer.

Example:

```json
{
  "type": "file-candidate",
  "to": "receiver-peer-id",
  "payload": {
    "file_id": "uuid",
    "candidate": "..."
  }
}
```

If the app reuses the meeting peer connection and data channel, this message may not be needed.

## Recommended Frontend Transfer Flow

1. Sender selects a file with `<input type="file">`.
2. Sender checks local frontend limits:
   - max size
   - allowed MIME types
   - current connection state
3. Sender creates or reuses a WebRTC data channel.
4. Sender sends `file-offer` through backend WebSocket.
5. Receiver shows an accept/reject prompt.
6. Receiver sends `file-answer`.
7. If accepted, sender chunks the file and writes chunks to the data channel.
8. Receiver reconstructs chunks into a `Blob`.
9. Receiver creates an object URL or saves via browser APIs.
10. Both sides clear transfer state after completion.

## Chunking Strategy

Recommended chunk size:

```text
16 KiB to 64 KiB
```

Example sender logic:

```js
const chunkSize = 16 * 1024;
let offset = 0;

while (offset < file.size) {
  if (channel.bufferedAmount > 1 * 1024 * 1024) {
    await waitForBufferedAmountLow(channel);
  }

  const chunk = file.slice(offset, offset + chunkSize);
  const buffer = await chunk.arrayBuffer();
  channel.send(buffer);
  offset += chunkSize;
}
```

Send a final metadata message over the data channel:

```json
{
  "type": "file-complete",
  "file_id": "uuid"
}
```

## DataChannel Message Format

A practical binary protocol:

1. Send a JSON metadata frame:

```json
{
  "type": "file-start",
  "file_id": "uuid",
  "name": "report.pdf",
  "size": 1234567,
  "mime": "application/pdf"
}
```

2. Send binary chunks as `ArrayBuffer`.
3. Send final JSON frame:

```json
{
  "type": "file-complete",
  "file_id": "uuid"
}
```

Alternative:

- Prefix each binary chunk with transfer ID and sequence number.
- This is better when multiple file transfers share one data channel.

## Progress And Backpressure

Sender progress:

```text
sent_bytes / file.size
```

Receiver progress:

```text
received_bytes / expected_size
```

Backpressure:

- Pause reading the file when `channel.bufferedAmount` is too high.
- Resume on `bufferedamountlow`.
- Keep memory bounded by not reading the whole file unless the size is small.

## Failure Handling

Frontend should handle:

- Receiver rejects transfer.
- Sender cancels transfer.
- Receiver cancels transfer.
- Peer leaves.
- Data channel closes.
- WebSocket disconnects.
- Browser memory pressure.
- File larger than frontend policy.

Recommended control frames over data channel:

```json
{
  "type": "file-cancel",
  "file_id": "uuid",
  "reason": "cancelled by sender"
}
```

## Transfer History Endpoint

The frontend can request recent transfer metadata:

```http
GET /api/rooms/{room_id}/files
```

Example response:

```json
[
  {
    "id": 1,
    "room_id": "room-id",
    "file_id": "uuid",
    "sender_peer_id": "peer-a",
    "target_peer_id": "peer-b",
    "name": "report.pdf",
    "size": 1234567,
    "mime": "application/pdf",
    "status": "offered",
    "ts": 1710000000
  }
]
```

History is metadata only. A refreshed client can show what was offered or accepted, but it cannot download the file from the backend.

```json
{
  "type": "file-error",
  "file_id": "uuid",
  "reason": "data channel closed"
}
```

## Security Considerations

Because file bytes do not pass through the backend:

- Backend cannot scan files.
- Backend cannot enforce file content policy.
- Backend cannot recover failed transfers.
- Backend cannot provide download links.

Frontend should:

- Show filename, size, and sender before accepting.
- Enforce maximum file size.
- Optionally restrict MIME types/extensions.
- Treat MIME type as untrusted user input.
- Avoid auto-opening received files.
- Revoke object URLs after use.
- Avoid storing files without user action.

## Relationship To P2P-First Media

File sharing follows the same philosophy as media:

- Use backend for coordination.
- Use browser-to-browser WebRTC for heavy bytes.
- Keep server bandwidth low by default.

If direct P2P data channels fail, deployment options include:

- TURN-assisted data channel relay.
- Server-upload fallback endpoint.
- Object storage fallback.

The current backend keeps the default path browser-to-browser and does not expose upload/download endpoints.

## Summary

Current file sharing is a signaling contract:

- Backend relays `file-offer`, `file-answer`, and `file-candidate`.
- Backend persists transfer metadata for history/audit.
- Frontend owns DataChannel creation, chunking, progress, cancellation, and file reconstruction.
- The server never stores or streams file bytes.
